CONTRACT ANALYSER

What this does

A user uploads a PDF contract and the app extracts the raw text and sends it to the Anthropic Claude API. Claude reads the contract and returns a validated JSON object containing the parties, governing law, contract term, key dates, obligations, a plain-English summary, and a list of risk clauses each rated high, medium, or low severity. The app then renders that JSON as a human-readable summary card and a colour-coded risk panel, and makes the same data available as a downloadable Word document.

How the pipeline works

  1. 01
    Text extraction

    pypdf reads the uploaded PDF and extracts the raw text content, page by page, into a single string.

  2. 02
    Structured extraction via Claude

    The text is sent to the Anthropic Claude API in a single API call. A strict system prompt instructs Claude to read the contract and return only a JSON object conforming to a defined schema — no prose, no markdown. Claude's only job is to read the contract and return structured data.

  3. 03
    JSON parsing

    Python parses the response with json.loads. If the response is not valid JSON the request fails explicitly rather than silently degrading.

  4. 04
    Rendering

    Flask passes the parsed dictionary to a Jinja2 template which renders two outputs from the same data: a structured summary panel on the left and a colour-coded risk clause panel on the right.

  5. 05
    Word document export

    The same JSON stored in the session is passed to python-docx, which writes it into a formatted .docx file served directly for download — demonstrating how a single extraction feeds multiple downstream output formats.

Why this architecture

Single API call rather than a conversation

Contract analysis is a stateless transformation. The full document is available upfront, the required output schema is fixed, and there is no iterative reasoning that benefits from multiple turns. A single call with a well-constructed system prompt is more predictable, cheaper, and easier to debug than a multi-turn agentic loop. Every run produces the same output format from the same input.

Structured JSON rather than free prose

Enforcing a strict schema in the system prompt means Claude's output is directly machine-readable without heuristics or post-processing. The app calls json.loads and either gets a clean dictionary or raises an exception — there is no ambiguous middle state. This also makes the output trivially portable: the same JSON renders in the browser, exports to Word, and could be written to a database or sent to a downstream API without any transformation layer.

Claude handles extraction only — Python handles everything else

Rendering, routing, file generation, and session management are all handled by Python. Claude is not asked to format output for display, make decisions about presentation, or produce anything other than data. This separation keeps the AI layer narrow and replaceable: the extraction prompt can be swapped or the model updated without touching the application logic, and the application logic can evolve without changing the prompt.
